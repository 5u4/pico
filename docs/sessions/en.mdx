This component owns the **live agent sessions** that the [Engine](carto:engine)
drives. Three concerns live here: building the shared SDK runtime once
(`runtime.ts`), pooling per-conversation sessions safely (`sessions.ts`), and
injecting pico's persona and the user's identity into every session's system
prompt (`identity.ts` + `persona.md`).

## The shared runtime

`provisionRuntime(options)` (`src/engine/runtime.ts`) builds one `OmpRuntime`
at startup and hands it to every session. Its `build()` (`src/engine/runtime.ts:68`):

- Initializes `Settings` and **forces `secrets.enabled = true`**
  (`src/engine/runtime.ts:75`) ŌĆö the runtime-level half of pico's
  secret-privacy duty (see [Secret guard](carto:secrets)).
- Refreshes the model registry `online-if-uncached`, **tolerating failure** with
  only a warning (`src/engine/runtime.ts:78-81`).
- Resolves a default model via a fallback chain: role selection `['default']` ŌåÆ
  first default-available ŌåÆ first available; errors only if none exist
  (`src/engine/runtime.ts:50-53`).

The whole function returns a `Result` ŌĆö a runtime that cannot resolve a model is
a hard boot failure (see the [boot sequence](carto:overview)).

## The session pool

`Sessions` (`src/engine/sessions.ts`) is the concrete `SessionsPort` the Engine
depends on. It keeps a `live` map of open `AgentSession`s keyed by
conversationId.

### Concurrency safety

`open()` must never build two sessions for the same conversation, and must not
leak a session opened during shutdown. Two mechanisms guard this:

- A `pending` promise map dedupes concurrent `open()` calls for the same id
  (`src/engine/sessions.ts:57-62`) ŌĆö the second caller awaits the first's promise.
- A monotonic `generation` counter (`src/engine/sessions.ts:39`) is captured at
  the start of an open; if `closeAll()` bumped it meanwhile, the freshly-built
  session is disposed and the open errors `sessions closed during open`
  (`src/engine/sessions.ts:73-76`).

### Constructing a session

`construct()` (`src/engine/sessions.ts:111`) is where a session comes to life:

- `mkdir ~/.pico/sessions/<id>`, then **resume the newest `*.jsonl` transcript**
  (via `latestSessionFile` ŌĆö filter `.jsonl`, sort, take last,
  `src/engine/sessions.ts:161`) or create a fresh one.
- `createAgentSession` with the shared runtime, `appendSystemPrompt =
  assembleAppendPrompt(loadIdentity(identityPath))`, and `extensions:
  [secretGuard]`.
- `autoApprove: true`, `hasUI: false` ŌĆö pico runs sessions **headless and
  unattended**; there is no human to approve tool calls in the loop.

## Identity: persona + user override

`assembleAppendPrompt(identity)` (`src/engine/identity.ts:28-30`) builds the
appended system prompt: `persona.md` alone, or `persona + "\n\n" + identity`
when the user supplies one.

- `persona.md` (`src/engine/persona.md`) is read **once at import**
  (`src/engine/identity.ts:9`). It defines pico as a warm, direct personal
  assistant and states two hard secret rules ŌĆö never ask the user to paste a
  secret, and if a plaintext secret is seen, warn the user to rotate it and
  don't echo it. These pair with the [secret guard](carto:secrets) enforcement.
- `loadIdentity(path)` (`src/engine/identity.ts:15-26`) reads an optional
  `~/.pico/identity.md`; a missing file returns `undefined` silently, other
  errors warn.

:::note
The persona's secret rules are a *prompt-level* promise. Enforcement is a
separate, non-negotiable layer ŌĆö the [secret guard](carto:secrets) extension
installed here ŌĆö because a prompt can be talked around and a hook cannot.
:::
