---
title: Persistence store
description: The WAL SQLite state database — open, migrations, query tracing, and the Zod row schemas.
---

The store is pico's durable memory. Two files: `db.ts` opens the SQLite
database, runs migrations, and wraps queries with tracing; `schema.ts` defines
the Zod schemas that are the **parse boundary** for every row read out of it.
The [Registry](carto:registry) is the only code that queries these tables
directly.

## Opening the database

`defaultDbPath()` is `~/.pico/state.db` (`src/store/db.ts:11-13`). `openDb(path)`
(`src/store/db.ts:14-23`):

- Creates the parent directory unless the path is `:memory:` (tests use the
  in-memory form).
- Opens with `{ create: true, strict: true }`, then sets two pragmas:
  `journal_mode = WAL` (concurrent reads during writes) and `foreign_keys = ON`
  (so the `ON DELETE CASCADE` from workspace to conversation is enforced).
- Runs `migrate(db)`, then wraps the handle with `traceDb` for query logging.

### Query tracing

`traceDb` / `traceStatement` `Proxy`-wrap the database's `query`/`prepare`/`run`
so every executed statement logs `{sql} {params} ({ms}ms)` at debug level via the
`db` logger (`src/store/db.ts:44-76`). `STATEMENT_EXECUTORS` names the methods
that actually run SQL (`run`, `get`, `all`, `values`, `iterate`,
`src/store/db.ts:25`). This is pure observability — it changes no results.

## Migrations

`MIGRATIONS` is an ordered list of `{ version, up }` (`src/store/db.ts:80-119`).
`migrate(db)` reads `PRAGMA user_version`, filters to migrations newer than it,
and applies the pending ones **inside a single transaction**, bumping
`user_version` after each (`src/store/db.ts:126-137`). `LATEST_SCHEMA_VERSION`
is the max version present (`src/store/db.ts:121`).

- **v1** creates `workspaces` and `conversations`, with a
  `CHECK ((defaultBranch IS NULL) = (branchPrefix IS NULL))` enforcing that a
  workspace is *either* fully worktree-configured or not at all, plus partial
  unique indexes on `(platform, externalId)` and `(workspaceId, externalId)`.
- **v2** drops the now-unused `conversations.branch` column.

:::note[Rapid-dev migrations]
Migrations here move forward only — v2 simply drops a column. Per the repo's
rapid-dev stance there is no down-migration or data-preservation path; the
maintainer rebuilds state by hand if needed.
:::

## Row schemas

`schema.ts` mirrors the table columns as Zod schemas so a row is validated the
moment it leaves the DB:

- `platformSchema = z.enum(['web', 'discord'])` (`src/store/schema.ts:3`).
- `workspaceSchema` / `Workspace` (`src/store/schema.ts:6-16`) — id, cwd,
  platform, nullable label/externalId, nullable `defaultBranch`/`branchPrefix`,
  `createdAt`.
- `conversationSchema` / `Conversation` (`src/store/schema.ts:24-34`) — note
  there is **no `branch` field**; it was dropped in migration v2.
- `isWorktreeWorkspace(workspace)` (`src/store/schema.ts:18-22`) — the type
  guard, true iff both `defaultBranch` and `branchPrefix` are non-null. This
  single predicate is what the [Engine](carto:engine),
  [Worktrees](carto:worktrees), and [WebHub](carto:web-hub) branch on to decide
  whether a conversation gets a git worktree.
