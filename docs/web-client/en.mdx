title: Web client
description: The React frontend — a pure reducer mirroring the wire protocol, bound to an auto-reconnecting WebSocket.
---

The client is the browser half of the [web transport](carto:web). Its design is
a clean split: `state.ts` is a **pure reducer** that is the mirror image of the
wire protocol, and `runtime.tsx` binds that reducer to a live, auto-reconnecting
WebSocket and to the `assistant-ui` React runtime. `connection.ts` holds the
reconnection constants; `persist.ts` restores the last conversation from
localStorage.

## Mental model: one reducer, one funnel

`reduce(state, action): { state, commands, toasts? }`
(`src/platforms/web/client/state.ts:315`) is the single funnel. Every UI action
and every inbound `ServerEvent` flows through it; it returns the next
`ThreadState`, a list of `ClientCommand`s for the runtime to **send**, and
optional toasts. The reducer is pure and framework-agnostic — `runtime.tsx`
executes its outputs.

### Folding server events

`reduceServer` (`src/platforms/web/client/state.ts:141-245`) handles the down
direction:

- `workspaces` — adopt `activeId` if present; else enter a fresh draft if a
  `draftWorkspaceId` is given; else just store the list (`:144-190`).
- `snapshot` / `older` / `stream` — **ignored unless
  `event.conversationId === state.activeId`** (`:191-231`), so frames for a
  conversation you just switched away from can't corrupt the view. Snapshots
  merge by id, `older` prepends a page, `stream` appends the latest.
- `attention` replaces the attention array; `heartbeatAck` is a no-op; `error`
  clears pending state and raises a toast (`:232-247`).

### Optimistic sends

`reducePrompt` (`src/platforms/web/client/state.ts:247-271`) sets an **optimistic
pending user message** and emits `create{prompt}` when in a draft, else
`prompt{text}`. `clearPendingIfResolved` (`:121-139`) drops that optimistic
message once the server's user-message count exceeds the captured baseline — so
the placeholder disappears exactly when the real message arrives, with no
flicker. Merge helpers `mergeTail` (`:60-71`) and `mergeById` (`:73-131`, ordered
by a numeric id suffix) keep the message list coherent across snapshot/stream/older.

## The runtime binding

`runtime.tsx` wires the reducer to the network and to React.

- **`dispatch(action)`** (`src/platforms/web/client/runtime.tsx:130-141`) =
  `reduce` → update a `stateRef` (synchronous reads inside socket callbacks) and
  React state → `flush` the returned commands via `send` → raise toasts.
- **`send`** JSON-stringifies a command to the socket if OPEN, else queues it in
  an `outbox` ref (`src/platforms/web/client/runtime.tsx:116-121`).

### Connection lifecycle

The socket is resilient (`src/platforms/web/client/runtime.tsx:143-263`):

- `connect()` opens `ws(s)://<host>/ws` (scheme from `location.protocol`).
- **On open**: reset backoff, replay the active conversation as a `select`, drain
  the outbox, and start a 15s heartbeat that closes the socket if no
  `heartbeatAck` arrives within the pong timeout.
- **On close**: stash the active id, set `reconnecting`, and schedule a
  reconnect with **jittered exponential backoff** (`backoffDelayMs` × 0.85–1.15,
  `connection.ts`).
- `wake()` on `online`/`focus`/`visibilitychange` eagerly revives a dead socket.

`connection.ts` supplies the constants — `RECONNECT_BASE_MS = 1000`,
`RECONNECT_MAX_MS = 15000`, `HEARTBEAT_INTERVAL_MS = 15000`,
`HEARTBEAT_TIMEOUT_MS = 10000` — and `backoffDelayMs(attempt)` =
`BASE × 2^attempt` capped at `MAX` (`src/platforms/web/client/connection.ts:13-17`).

### Persistence and bootstrap

`persist.ts` reads/writes localStorage **through Zod**: every `readPersisted`
re-validates against a schema and falls back safely, so stale or corrupt storage
never crashes the app (`src/platforms/web/client/persist.ts:10-23`). On first
`workspaces` with no active conversation, the runtime restores the persisted
`activeConversation` if it still exists, else creates one in the first workspace
(`src/platforms/web/client/runtime.tsx:266-290`). It also sets the document title
to `(n) pico` from the attention count (`:296-299`).

### assistant-ui

`AssistantPane` wires `useExternalStoreRuntime`, mapping engine `Message` parts
to assistant-ui messages (special-casing `tool-call`), routing `onNew → prompt`
and `onCancel → cancel`, and keying the thread by `threadKey` so switching
conversations resets it cleanly.
